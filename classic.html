Seedance 2.0、結局どこで使えばいいの？——プラットフォーム・コスト・注意点を全部調べた

正直に言うと、Seedance 2.0はあまり触れていなかった。

3月末から各プラットフォームで使えるようになって、タイムラインが一気にSeedance一色になったのは見ていた。アニメバトルの動画がバズっていたし、「ハリウッド終わった」みたいな投稿も流れてきた。でも自分はKlingやViduを中心に回していたので、ちゃんと触る前にまず「全体像を把握したい」と思った。

で、1週間分のX投稿を片っ端から調べて、さらに公式ドキュメントやAPI料金も深掘り調査した。

バズ投稿、ユーザーの不満、料金体系、プラットフォームの違い、Tips、他ツールとの比較——全部まとめたのがこの記事。「Seedance 2.0、気になってるけど何から始めればいいかわからない」という人向けに書いた。

まず知っておくべきこと

Seedance 2.0はByteDance（字節跳動）が開発したAI動画生成モデル。 Higgsfieldやfreepikは「Seedance 2.0を自社プラットフォームで使えるようにしているサードパーティ」であって、開発元ではない。

  • 開発元: ByteDance
  • リリース: 2026年2月10日（ByteDance発表）
  • 各プラットフォーム展開: 2026年3月末〜4月初旬にかけて順次

2.0の主な進化ポイント

マルチモーダル参照

画像・動画・音声を同時に参照素材として入力できる。具体的な上限:

  • 画像: 最大9枚（各30MBまで）
  • 動画: 最大3つ（各50MBまで）
  • 音声: 最大3つ（各15MBまで）
  • 合計: 最大12ファイル

プロンプト内で@Image 1、@Video 1のようにタグ指定して、各素材の役割をディレクションできる。

キャラクター一貫性

同一キャラが異なるシーン間を移動してもドリフト（顔崩壊）しない。OpenArtでのテスト投稿では「ドリフトゼロ」と報告されていて、ジャケットの刺繍やラベルのタイポグラフィまで15秒間保持される。

物理演算

水流、火炎、衝撃波、破壊エフェクトがリアルに動く。特にアニメバトル系のクオリティが飛び抜けている。Sasuke vs Muzanの戦闘動画は1,521いいね/470保存、Superman vs Darkseidは524いいね/205保存を記録していた。

ネイティブオーディオ生成

映像と音声を並列処理で同時生成（後付けではない）。8言語以上で音素レベルのリップシンクに対応。環境音・効果音・音楽もプロンプトから同時生成される。グラスが割れる映像には、空間に合わせたリバーブ付きの破砕音が自動で付く。

解像度・尺

  • 解像度: 最大2K（2560×1440）、1080p、720p
  • 尺: 4秒〜最大15秒
  • アスペクト比: 16:9、4:3、1:1、3:4、9:16 等

知っておくべき注意点

1. 日本ユーザーの利用制限——著作権問題が背景

日本と米国からのアクセスが制限されている理由は、ハリウッドのスタジオ（ディズニー、パラマウント等）からの著作権侵害の抗議。ByteDanceがトレーニングデータにこれらの著作物を無断使用した疑いが指摘されており、法的リスク回避のため知的財産保護に厳しい法域（日本・米国）での展開を制限している。

具体的な制限:

  • Higgsfield、Invideo等: 日本からのアクセスは原則ブロック（Business認証が必要）
  • Dreamina: 日本からの通常アクセスにはVPNが必要になる場合がある
  • 顔リファレンス: 実在人物の顔写真のアップロードはシステムレベルでブロック（HeyGen以外）。AIが自動検出して生成拒否する
  • 暴力・政治的テーマ: フィルタリング対象

日本語圏のクリエイターからの詳細な規制レポート（❤️156/🔖46）でも、「Klingに戻る」という声が上がっていた。

Claude Codeの利用枠が尽きても止まらない — 中身をGPT-5.6 Solに差し替える『claudex』を実機構築した

Claude Codeを毎日使うようになってから、月の途中で「利用枠が切れました」の表示を見る回数が増えた。作業中のセッションがそこで止まる。続きは翌日の枠が回復するまで待つしかない。

僕の場合、同じタイミングでChatGPTのサブスクも払っている。Codex CLIも入れてある。枠が切れているのはClaude側だけで、ChatGPT側の枠はまだ残っている日がほとんどだった。だったら、Claude Codeの「体」はそのままに、中身の頭脳だけChatGPT側に差し替えて動かせないか。そう思って、X上で話題になっていたTibo (@thsottiaux) とTheo (@theo) のセットアップガイドを頼りに、実機で組んでみた。

結果から言うと、動いた。正確には、動いたのは半分だ。Codex CLIから直接GPT-5.6 Solを叩く経路は、セッションログの裏取りまで含めて実測で確認できている。一方でclaudex経由の経路は、プロキシの起動と認証フローの開始までは実機で確認できたが、そこから先「claudex経由でSolが実際に応答する」最後の一歩は、この記事を書いている時点ではまだ元ガイドの報告に依拠している。どこまでが自分の目で見た事実で、どこからが又聞きか。それを分けて書く。

結論: 3経路の使い分け地図

今、手元にある経路は3つ。

普段のClaude Codeは、Anthropicに直接つながっている。頭脳はClaude、課金はAnthropicの枠、MCPはコネクタもローカル定義も全部使える。ここが本体で、迷ったらここに戻ればいい。

claudexは、Claude Codeというハーネスはそのままに、頭脳だけGPT-5.6 Solに挿げ替えたもの。課金はChatGPTサブスクの枠を消費する。MCPはローカル定義のみで、claude.aiのコネクタはそのセッションだけ無効になる。認証がプロキシ側に上書きされるためだ。スキルもhookもエージェント定義も、同じ.claudeを読むのでそのまま使える。ここが一番の強みで、Claude枠が切れてもClaude Code運用を丸ごと続けられる。

Codex CLIは、また別のハーネスだ。頭脳はGPT-5.6系(solやterraなど切り替え可能)、課金はChatGPTサブスク枠、MCPも実は使える(これは後で誤解として書く)。ただしスキルやhookやサブエージェントといったClaude Code資産は引き継げない。AGENTS.mdやpromptsという独自の系統で動いている。

3つとも「頭脳」と「体」の組み合わせが違うだけで、それぞれ得意な場面がある。

claudexの仕組み、ローカル翻訳プロキシという発想

claudexは新しいツールではない。ローカルに立てた翻訳プロキシに、Claude Codeの向き先を変えるだけの仕組みだ。

claudexというaliasを叩くと、内部ではClaude Code CLIがそのまま起動する。ただし通信先はAnthropicではなく、localhost:8317で待ち受けているCLIProxyAPIだ。このプロキシが、Anthropic Messages API形式で届いたリクエストをOpenAI Codexバックエンドが読める形式に翻訳して転送し、返ってきた応答をまたAnthropic形式に戻してClaude Codeに渡す。

Claude Code自身は、自分がAnthropicと話していると思ったまま動いている。実際に会話している相手はOpenAIのCodexバックエンドで、しかも自分のOAuthトークンで、というのがこの仕組みの核心だ。

プロキシの実体はCLIProxyAPI v7.2.86(2026-07-17リリース、MIT、github.com/router-for-me/CLIProxyAPI)。macOS向けの元ガイドは、Homebrewで導入する7ステップの手順を公開していた。

セットアップ完全手順(macOS 10分・WSL2読み替え実測)

まずmacOSの元ガイド通りの手順から。

brew install cliproxyapi
openssl rand -hex 24

生成した鍵を/opt/homebrew/etc/cliproxyapi.confに書き込む。host: "127.0.0.1"を指定し、api-keys:に先の鍵を入れ、payload.overrideでreasoning.effort: highを設定する。

brew services start cliproxyapi
cliproxyapi --codex-login
curl http://127.0.0.1:8317/v1/models

--codex-loginはブラウザでauth.openai.comの認証画面を開く。認証が終わると~/.cli-proxy-api/にトークンが保存され、以後は自動でリフレッシュされる。/v1/modelsを叩いてgpt-5.6-solが(gpt-5.6-lunaやgpt-5.5と並んで)見えていれば、プロキシ側は準備完了だ。

あとは~/.zshrcにaliasを追記する。env変数は7種類あって、それぞれ役割が違う。

alias claudex='ANTHROPIC_BASE_URL=http://127.0.0.1:8317 \
  ANTHROPIC_AUTH_TOKEN=<生成した鍵> \
  ANTHROPIC_DEFAULT_HAIKU_MODEL=gpt-5.6-sol \
  CLAUDE_CODE_SUBAGENT_MODEL=gpt-5.6-sol \
  CLAUDE_CODE_ALWAYS_ENABLE_EFFORT=1 \
  CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY=3 \
  ENABLE_TOOL_SEARCH=false \
  claude'
  • ANTHROPIC_BASE_URL: Claude Codeの向き先をローカルプロキシへ変える
  • ANTHROPIC_AUTH_TOKEN: プロキシへの認証。自分で作ったローカル鍵で、OpenAIのキーではない
  • ANTHROPIC_DEFAULT_HAIKU_MODEL: 小型・背景モデル呼び出しもSolへ回し、Anthropicに漏れないようにする
  • CLAUDE_CODE_SUBAGENT_MODEL: サブエージェントも同じモデルで動かす
  • CLAUDE_CODE_ALWAYS_ENABLE_EFFORT: 非Anthropicモデルでもreasoning effortの制御を有効化する
  • CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY: 並列ツール呼び出しを3本までに制限する。Codexバックエンドはバースト耐性が低い
  • ENABLE_TOOL_SEARCH: 遅延ツール発見は非Claudeモデルだと不安定なので無効化する

これはaliasの中にインラインで並べているだけなので、この起動のときだけ適用される。素のclaudeを叩いたときは従来どおりAnthropicに直接つながる。claudex --continueやclaudex -p "..."のように、引数はそのまま透過する。あとはスモークテストとしてclaudex -p "..."を叩けば、元ガイドの手順は終わりだ。ここまでで実測10分、というのが元ガイドの主張になる。

僕の環境はWSL2なので、ここから先は読み替えが必要だった。Homebrewは使えないので、GitHubリリースからCLIProxyAPI_7.2.86_linux_amd64.tar.gzを落として~/apps/cliproxyapi/に展開する。configは~/.cli-proxy-api/config.yamlに置くのがLinux推奨パスだ。

常駐はlaunchdの代わりにsystemdのユーザーサービスを使う。

~/.config/systemd/user/cliproxyapi.service

[Service]
ExecStart=%h/apps/cliproxyapi/cli-proxy-api --config %h/.cli-proxy-api/config.yaml
systemctl --user enable --now cliproxyapi
loginctl enable-linger

aliasは~/.bashrcに書く。OAuthログインは--codex-login --no-browserでURLだけ表示させ、Windows側のブラウザで開く。コールバック先のlocalhost:1455は、WSL2のポートフォワードで問題なく届いた。実測でも1455の待受は確認できている。

ログイン前に/v1/modelsを叩くと{"data":[],"object":"list"}という空の応答が返ってくる。これが「まだログインが終わっていない」のサインだった。最初これを見たときは設定ミスかと疑ってconfig.yamlを何度も見直したが、単にログインが先という順番の問題だった。

実機で起きたつまずき2つ

手順通りに進めたつもりでも、2箇所でつまずいた。ここは隠さず書く。

ひとつめは、Codex側のプロファイル仕様変更だ。元ガイドや古い記事はconfig.tomlに[profiles.sol]と書く前提で説明しているが、手元のcodex-cli 0.144.4ではこれがそのままエラーになった。

--profile `sol` cannot be used while config.toml contains legacy `profile = "sol"` or `[profiles.sol]` config; move those settings into ~/.codex/sol.config.toml

正解は、プロファイルを別ファイルに分離すること。~/.codex/sol.config.tomlにmodel = "gpt-5.6-sol"などを書き、codex --profile solやcodex exec --profile sol "..."で起動する。旧記法の記事を参考にしていると、ここで必ず止まる。

もうひとつは、CLIProxyAPIのOAuth待受のタイムアウトだ。--codex-loginを叩いてから認証URLをクリックするまで、約5分の制限時間がある。14時50分にコマンドを打って、他の作業を挟んでから14時55分にURLをクリックしたら「Authentication timed out. Please try again.」で失敗した。

このURLはPKCEの使い捨てトークンなので、失効したら再発行しないといけない。先にURLだけ発行しておいて、後でゆっくりクリックする、というのは通用しない。ブラウザの前に座った状態でログインコマンドを打ち、出てきたURLを5分以内に開く。それだけの話だが、知らずに1回無駄にした。

正直に書いておくと、claudex -p "..."によるフルのスモークテストは、この記事を書いている時点ではOAuthログインの完了待ちで実施できていない。プロキシの起動、モデル一覧の空応答、OAuthフローの開始までは実機で確認できたが、claudex経由でGPT-5.6 Solが実際に応答するところまでは、元ガイドの報告に依拠した記述になる。ここは断定しない。

一方で、GPT-5.6 SolがCodex CLI単体で現役なのは、こちらは実測で確認済みだ。codex exec -m gpt-5.6-sol --skip-git-repo-check "..."は2026-07-18に正常応答し、~/.codex/sessions/2026/07/18/rollout-*.jsonlに"model":"gpt-5.6-sol"という記録も残っていた。ちなみに2026-07-10の時点ではcodex-cli 0.144.1で実走に成功していたが、CLI 0.125時点ではAPI側がバージョンを見て拒否していた。この手のバージョン要件は、公式ドキュメントより実際に叩いてみたほうが早い。

よくある誤解: Codex CLIでもMCPは使える

ここで一つ訂正しておきたい誤解がある。「Codex CLIはMCPに対応していない」というものだ。

これは間違いだった。config.tomlに[mcp_servers]を書けば、stdioでもURLでも普通にMCPサーバーをつなげられる。手元の環境では、実際にCodex CLI側へ150本のMCPを登録して動かしている実例もある。

じゃあclaude・claudex・Codex CLIの本当の違いはどこにあるのか。MCPの有無ではない。ハーネス層、つまりスキルやhookやサブエージェント定義やメモリの有無だ。claudeとclaudexは同じ.claudeディレクトリを読むので、この資産をそのまま使い回せる。Codex CLIは別ハーネスで、AGENTS.mdやpromptsという独自の仕組みで動いている。MCPという道具箱は3経路とも似たものが使えるが、その道具箱をどう使うかの手順書が違う、というのが正確な理解だと思う。

デメリット3点を正直に

claudexを触ってみて、いいことばかりではないというのも書いておく。

ひとつめは規約のグレーさ。CLIProxyAPIはCodex CLIになりすます非公式クライアントで、OpenAIが規約違反と判断すればアカウント制限を受ける理論的なリスクがある。現状は多くのユーザーが使っていて黙認されている状態だが、ChatGPTのアカウントは他の用途にも使っているインフラなので、常用はせず「枠切れ時の非常口」に留めておくのが安全だと思う。

ふたつめはサブスク枠の共食い。claudexで消費する分は、Codexの5時間枠や週間枠と同じ財布から出ていく。長いセッションを回すと、Codex CLI側で使いたかった別の用途、たとえば画像生成の枠が先に切れてしまうことがあり得る。

みっつめは互換性の粗さ。翻訳プロキシを経由する分、ツール呼び出しが本家より不安定になりやすい。だからこそ元ガイドは並列ツール呼び出しを3本までに絞り、遅延ツール発見(Tool Search)を無効にすることを勧めている。claude.aiのコネクタMCPが使えないのも、この互換性の粗さの一部だと考えていい。

放置コストは低い。localhost限定でbindして、鍵認証をかけて、管理APIを無効にしておけば、常駐リソースは数十MB程度で、素のclaudeには影響しない。撤収するときはsystemctl --user disable --now cliproxyapiとaliasの削除だけで済む。外に出る資格情報もOpenAIのOAuthトークンがOpenAIに行くだけで、元ガイドの設計としては閉じている。

使い分けの結論

ここまで組んでみて思うのは、claudexは常用のツールではなく、枠が切れた日だけ開ける非常口として運用するのがちょうどいい、ということだ。

GPT-5.6 Solの頭脳としての実力は、別の場面で確かめている。2026-07-10、30秒のティザー映像の監督業を丸ごとSolに任せたことがあった。構成、映像プロンプト、音楽、編集指示、リテイクの判断まで。Fable 5版との競作という形だったが、失敗の原因を特定して自分のプロンプトを精密化していく「自己改訂」の動きが印象的だった。編集契約書の数値を実測値に合わせて自分で修正してくるところも、監督としては実用域だと感じた。

だから、claudexという経路そのものへの信頼は薄くない。頭脳の実力は知っている。ただ、その頭脳をClaude Codeの体に載せ替える経路には、規約のグレーさと互換性の粗さという別の重さがある。

普段はclaude。枠が切れて、その日中にどうしても進めたい作業があるときだけclaudex。MCPを使いながら軽く動かしたいときや、Claudeとは違う視点でクロスチェックしたいときはCodex CLI。3つを頭の中の地図として持っておけば、少なくとも「枠切れで作業が止まる」という一番困る瞬間には、選択肢が残っている。

もし試すなら、いきなりclaudexから組まなくてもいいと思う。まずはcodex exec -m gpt-5.6-solを1行叩いて、Solの返しがどんな感じか確かめるだけなら5分もかからないし、追加の課金も発生しない。頭脳の感触はそこで先につかんでおける。

claudexの本体を組むときは、OAuthの5分制限だけ覚えておいてほしい。--codex-loginを打つ前にブラウザの前に座っておく。それだけで、僕が無駄にした1回分は再現しなくて済む。

実際に組んでみて、動いた・動かなかったがあれば、コメントで教えてもらえるとうれしい。特にmacOS側でのフルスモークテストがどう転んだかは、こちらもまだ追いきれていないところなので気になっている。

まず地図を持つ ― 3つの型

比較を始める前に、この3分類だけ頭に入れておくと迷わなくなる。

「エージェント型」は、ブリーフやプロンプトを1つ入れると、台本・絵作り・音声・編集まで自動で回って完成動画が出てくるタイプ。「モデル直販型」は、映像を生成するモデルそのものへのアクセスが主で、台本〜編集の組み立てまではやってくれない、いわば素材生成ツール。「編集支援型」は、すでにある素材のクリップ化・字幕・整形など「加工」を自動化するタイプ。同じ「AI動画サービス」という言葉でくくられていても、この3つがやっていることはまったく違う。

比較を難しくしているのはここだと思う。エージェント型とモデル直販型を同じ土俵で料金比較すると、素材しか作らないサービスが不当に高く見えたり、逆に台本まで自動で組んでくれるサービスの価値が伝わらなかったりする。まず「自分が代行してほしいのはどの工程か」を決めてから、その型の中で比べるのが近道になる。

台本〜完パケまで担うのはエージェント型だけ。比べる前に、委ねたい工程を先に決める。

主役はエージェント型 ― 2026年7月時点の全図鑑

ここからが本題。「ブリーフや台本を渡すと完成動画が返ってくる」を掲げるエージェント型サービスを、地域も規模もばらばらに並べてみる。まず1つだけ厚めに紹介してから、残りは表にまとめる。

読者の入口になっているのはおそらくOiiOiiだ。チャット画面に台本の概要とキャラクターの参考画像を渡すと、Art Director・Screenwriter・Character Designer・Scene Designer・Storyboard Artist・Animator・Sound Engineerという7人の専門エージェントが役割分担しながら世界観・脚本・キャラ・絵コンテ・動画・音楽まで組み上げていく。Simple Modeなら約10分で完成品まで到達し、人間の作業は最初のプロンプト入力と最後の確認だけで済む。工程を自分の意図で直したい場合はProject Modeに切り替えると、セリフの言い回しやショットの組み方まで対話しながら30〜60分かけて詰められる。ただし正直に書いておくと、料金は公式サイトに一切の金額が載っていない。サードパーティのレビューサイトが揃って挙げている推定値は月$9前後から$169前後まで幅があり、2分程度のアニメ1本でおよそ$20前後かかるという試算も複数の記事が独自に出しているだけで、公式の単価表は存在しない。運営会社も一次情報では特定できていない。体験の良さと情報の不透明さが同居しているサービス、というのが今の実態に近いと思う。

ここまで調べて見えてきた共通の真実がひとつある。完全に無人でワンショット、という段階のサービスはまだ無く、どれも実態は「エージェントが下ごしらえをして、人間が最後に承認する」半自動だという点だ。PixVerse Agentのように「最終的な編集判断は人間」と自分で明記しているサービスもあるし、OiiOiiのSimple Modeのように承認だけで済む例もあるが、程度の差はあれ人間の目が入ることが前提になっている。

そもそも料金体系の多くは「クレジット」というサービスごとの内部通貨を消費するプリペイド制で、プラン名や月額の数字だけを見ても実際に何本作れるかはピンとこない。たとえば「60クレジットで10秒の動画1本」のような感覚で消費されるが、この換算レートはサービスごとに全部違う。表の「自動化の範囲」は、台本の自動生成・複数ショットへの自動分解・モデルの自動選択・音声の自動統合・最終つなぎ込みの自動化という5つの工程のうち、どこまで人手なしでこなせるかで判断した。同じ「AIエージェント動画サービス」を名乗っていても、この5つのうち何個を満たすかでできることの幅がかなり違う。料金は各社の公式表示の通貨のまま並べ(日本発サービスは円、他はドルのまま)、公式に確認が取れていない数字には記号を付けた(意味は表の下)。2026年7月11日時点の情報で、この分野は変動が速いので契約前に必ず公式ページで再確認してほしい。

どのサービスも越えられない3つの壁、そして1つの前提

壁は使うほど響く。特に③の従量の崖は量産フェーズで一気に効いてくる

どのサービスも便利だけど、工程を自分で組む側の視点で見ると、共通してぶつかる壁が3つある。

1つ目は「工程のカスタム不可」。台本の型・カメラワーク・尺の感覚を自分色に変えたいと思っても、サービス側が用意したワークフローの枠から出られない。エージェント型は特にここが強く、「ブリーフを入れれば勝手にいい感じにしてくれる」の裏返しとして、細部の演出意図を差し込む余地が狭い。

2つ目は「資産が積み上がらない」。毎月払い続けても、手元に残るのは「生成された動画」だけ。台本の型・背景素材・編集の型といった、次に使い回せる部品は資産化されない。解約したら振り出しに戻るし、サービスを乗り換えるたびに学び直しも発生する。

3つ目は「量産時の従量の崖」。1本だけ作るなら快適でも、10本20本と量産フェーズに入るとクレジットが溶ける速度が跳ね上がる。HeyGenのAvatar IV機能は1分あたり20クレジットを消費するとされ、Creatorプラン(月200クレジット)だと月10分ぶんで使い切る計算になるという解説記事の試算がある。Xで見かけた「オープンソースの組み合わせで$33〜124/月のスタックを組んで、SaaSの上乗せ分を避けた」という投稿も、この崖を避けるための工夫として読める。

そして壁とは別に、前提がもう1つ。サービスが今のまま存在し続ける保証はない。ChatGPT内のSoraは2026年4月にWeb/アプリの単体提供を終了した(出典: OpenAI Help Center)。1日の運用コストが売上を大きく上回っていたことが背景とする海外報道もある。Lumaも2026年3月に単体モデルツールから「Luma Agents」へ大きく舵を切った(出典: Luma公式X告知)。どちらも規模の大きい会社の判断で、個人が契約している側からすると「昨日まで使えていた機能が今日は無い」という形でしか変化に気づけない。サブスクは「今日ある機能が来月もある」ことまでは保証してくれない。

「組む」の実物 ― 私が実際に測った数字

ここからは私が実際に組んで測った数字の話。私は「毎朝、Xのトレンドを調べて、内容を精査して、1本の解説動画に仕立てて、投稿文と承認待ちのカードまで用意する」という一連の流れを、一度だけ設計して、あとは無人で回る仕組みとして自分で組んでみた。

実測すると、1本あたりの制作時間は約60分。人の作業はゼロ。動画生成のコストは、背景を2クリップ使うケースで合計120クレジットほど(無料残高の範囲内なら追加の支出はゼロ)。ナレーションはTTSの無料枠、動画の合成・テロップ入れはローカルで動かしているコード(React系の動画編集フレームワーク)なので、ここも0円だった。

別の実測として、動画生成モデルのtext-to-videoは10秒・720pで60クレジット、生成時間は実測で70秒前後だった。

ただ、ここまで滑らかに書くと嘘っぽくなるので正直に付け加えておく。仕組みを組んでいる最中、動画の最終書き出しをWindows側のマウントフォルダに直接向けて実行したら、書き出したはずの映像データが壊れていて再生できないファイルが並んでいたことがあった。書き出し先をLinux側のホームディレクトリに変え、書き出し後に自動でデコード検証を挟む仕組みを足すまで、夜の作業時間が数時間溶けた。もうひとつ、生成結果が意図とズレたときに禁止事項を長々書き並べた縛りの強いプロンプトでリトライを重ね、クレジットを無駄撃ちしたこともある。結局いちばん使えたのは最初に自由に生成させた1本で、そこで学んだのは「リトライの前に一度立ち止まる」ということだった。

数字そのものより大事なのは、「仕組みを一度組んだあとの限界コスト」の話だと思っている。背景素材を作ってストックしておき、翌日以降はそれをローテーションで使い回すようにすると、2日目以降の追加コストはほぼゼロに近づいていく。サブスクは毎月同じ金額が消えていく設計だけど、自分で組んだ工程は「一度作った部品」がそのまま資産として残り、使うたびにコストが下がっていく。

肝はループになっていること。仕組みを一度組めば、翌日以降は人が触らずに回り続ける

サービスでいい人、組むべき人

分かれ目は量産フェーズに入っているかどうか。100本作る頃に「型」の有無が効いてくる

ここまでの内容を踏まえて、自分がどちら側かを考えてみてほしい。

サービスで十分なのは、週1本ペースで満足していて、やりたいことが研修動画やアバター解説みたいな「型ど真ん中」にハマる人。あとは単純に、自分の時給で考えたら契約した方が安い人もこちら側。この場合は無理に組まず、地図に沿って合う型を1つ選ぶのが早い。台本を書けば多言語版まで自動で出てくるような型ど真ん中の用途なら、カスタム性を我慢するコストより恩恵の方がずっと大きい。

組むべき人は、量産フェーズに入っている人。台本の書き方・画角・テロップの入れ方といった自分の「型」を持ちたい人。そして将来、工程そのものを商品として売っていきたい人。この場合、サブスクの月額はいずれ「もったいない」に変わっていく。1本あたりで見れば安くても、100本作る頃に「型を持たないまま同じ額を払い続けている」ことに気づく瞬間が来る。

最後に

私自身は後者を選んで、コーディングエージェントと動画生成のMCP、動画編集フレームワークを組み合わせて工程を作った。最初の一歩は具体的で、動画生成モデルを1つ、APIかMCP経由で呼んで10秒のクリップを1本出すこと(MCPが何か分からなくても大丈夫。要は「コーディングエージェントから動画生成サービスを呼び出すための接続口」くらいの理解でいい)。本当にそれだけだった。台本の型やテロップの部品は、その後に少しずつ育てていけばいい。

サブスクをやめろという話ではない。週1本のペースで満足しているなら、地図の中から合う型を選んで契約するのが一番早い。ただ、量産したい欲や「自分の型を持ちたい」という気持ちが少しでもあるなら、工程そのものを組む選択肢があることは知っておいて損はないと思う。

AI映像ができたら勝手にGoogle Driveに入る、を作った話 — Claude Code × rclone で配信レイヤを自動化

「動画が完成したよ、Driveに入れといたから見てね」

この一言を毎回手動でやるのが、地味に効いてくる。AI映像制作のパイプラインを作り込んでいくと、ストーリー設計から編集まで全部自動化できるのに、最後の「Driveにアップする」だけが手作業で残る。一本あたりは大した時間じゃない。でも一日に何本も回すと、これが思考のリズムを切る。

しかも厄介なのは、配信先が一本道じゃないこと。

「これ確認してほしい」のレビュー段階でも Drive に置きたいし、品質ゲートを通った正本も Drive に置きたい。でもこの 2 つは混ぜたくない。レビュー版は何度もリテイクが回るから版数管理が必要。正本は逆に「常に最新の一本」だけが分かりやすい。

ということで、AI映像パイプラインの最終段に 配信レイヤ を一個挟んで、これを自動化した。今日はその設計と実装の話。

何を解決したかったか

うちの映像制作は 5 つのワークフローに分かれている。

  • 短尺クリップ集や MV の seedance-to-vidu
  • 60秒の物語を作る multishot-narrative-60s
  • CM・短尺広告の cm-pipeline
  • 企業PR映像の pr-video-pipeline
  • 漫画原作からアニメ化する manga-to-anime-pipeline

どれも Phase 8 で Remotion 編集が終わって final.mp4 が出来上がり、Phase 8.5 の品質ゲートを通って完成、という構造になっている。この「完成」の定義がワークフローによって微妙に違うのに、配信先が同じ Google Drive なのは共通している。

つまり、配信レイヤは WF を超えて共通の薄い層 として切り出すのが筋がいい。

設計の前に潰した選択肢

最初に考えたのは、Claude Code には Google Drive の MCP ツールが既に入っているので、それを使えば一行で書けるんじゃないか、という案だった。

ところがこれが致命的にダメだった。

MCP の create_file は content フィールドが base64 エンコード必須。60秒・4K の動画は 100MB から 500MB ある。base64 化すると 33% 膨張するから、500MB が 665MB になる。これを単一の MCP ペイロードで送るのは、タイムアウトの前に上限に引っかかる。短尺の Quick モードならギリギリ通るかもしれないけど、Pro モードで Topaz アップスケールまで通した動画は問答無用で死ぬ。

ということで MCP は早々に脱落。残った選択肢を表にすると:

案大容量リジューム認証の手間採用
rclone CLI✅✅OAuth 1回★
gdrive CLI (glotlabs)✅✅OAuth 1回△
Google Drive API を Python で直叩き✅✅ resumable uploadサービスアカウント設定✕
MCP create_file (base64)✕ ~50MB が限界✕既設定✕

決め手は、すでに Cloudflare R2 のために rclone を運用していたこと。r2: リモートの設定がすでに ~/.config/rclone/rclone.conf に入っているので、ここに gdrive: を追加すれば一本化できる。新しいツールを増やさない。

WSL ユーザーが踏みやすい罠

rclone config で Google Drive リモートを追加するとき、対話の中盤で Use auto config? と聞かれる。これは「ローカルでブラウザを起動して OAuth するか」というオプション。デフォルトは Yes。

WSL でこれを Yes にしてはいけない。

WSL の中から Windows のブラウザを起動できないことが多くて、結果として トークンが空のまま remote だけが保存される。次に rclone about gdrive: を叩くと empty token found エラーで弾かれる。1回目これで詰まった。

正解は Use auto config? = n。すると rclone が認証 URL を表示して入力待ちになる。この URL を Windows 側の普通のブラウザで開いて、Google アカウントで認証して、画面に表示されるコードをコピペで返す。これなら確実に通る。

ちなみに OAuth の途中で「このアプリは確認されていません」が出るけど、これは rclone の標準クライアントが Google の審査を受けていないだけで、ローカルツールとしては安全。「詳細」→「安全でないページに移動」で進める。

ディレクトリ構造は2段に固定した

Drive 側はこうした:

gdrive:AI_Video_Library/
├── Reviews/
└── Final/

最初は AI_Video_Library/Reviews/MV/, AI_Video_Library/Reviews/CM/, ... とジャンル別に切ろうかと思った。でもやめた。

理由は、5 つの WF の境界が思ったよりファジーだから。pr-video-pipeline のアニメモードと manga-to-anime-pipeline は実質同じ表現になる。seedance-to-vidu で MV を作っても CM みたいな尺になることがある。境界を切るほどメンテ負債になって、最後は誰も従わなくなる。

それより 「確認用」と「完成版」の 2 軸に絞る ほうが、人間にもスクリプトにも分かりやすい。ファイル名にプロジェクト slug (260502_short_xxx) を埋め込んでおけば、後から grep で逆引きできる。

命名衝突の処理

確認用は何度もリテイクが回るので、衝突しないように版数を自動でつけることにした。

N=1
EXISTING="$(rclone lsf gdrive:AI_Video_Library/Reviews/ 2>/dev/null || true)"
while echo "$EXISTING" | grep -qx "${PROJECT_NAME}_v${N}.mp4"; do
    N=$((N+1))
done

愚直。rclone lsf でフラットに一覧取って、grep で完全一致をチェックして、空いている番号を探す。最初の v1 から順に上がっていく。

完成版は逆に、版数なし <project>.mp4 で固定にして、上書き運用にした。「常に最新の一本だけが正しい」という運用ルールが明確になる。

5 つの WF への組み込み

各 WF の SKILL.md を全部書き換えるのは避けたかった。Phase 8 のロジックは WF ごとに違うし、Phase 8.5 の中身も法務QC だったり品質ゲートだったり様々だから、そこに手を入れると相互独立が壊れる。

## Phase 9: Distribution — Google Drive 自動アップロード

### 9.2 Phase 8.5 PASS 直後 → 完成版

5 ファイルに同じブロックを入れる作業は数分で終わる。WF 個別のロジックには一切触っていない。

完成して何が変わったか

実用テストはまだだけど、スモークテストでは:

  • 確認用アップロード → 自動で _v1.mp4, _v2.mp4 と版数が振られた
  • 完成版アップロード → 上書きで正本が更新された
  • 共有リンクが返ってくるので、そのまま「これ見て」とコピペできる

スクリプトは 60 行ちょっと。MCP を使わずに、すでにある rclone を活用しただけ。新しい依存はゼロ。

何より、自分の頭から「Driveにアップする」というタスクが消えた。これが一番効いている気がする。映像が出来た瞬間、すでに Drive に入っていて、URL が手元にある。次の作業に意識を移せる。

やりたくなるでもやらなかった理由

本文 (貼るだけ)

去年の自分は、キャラを動かす方法が多すぎて、どれを選べばいいのか全然わからなかった。

テキストから作るAI、動画から動きを抜くやつ、Mixamoみたいな既製品、Blenderで手付け。名前は聞くけど、結局どれがいくらで、何が得意なのか。並べて説明してくれる人がいなかった。

なので今の自分が、7つ全部並べて「料金」と「手間」で1枚の地図にしてみた。

これだけ掴めば、全部わかる

基準になるのは「Blenderで1から手付けするプレビズ」。ここでいうプレビズは、完成前に動きの流れを確認する下書き映像のこと。これが 制御は最大・お金は0円・でも手間が最大 の極にある。思い通りに何でも作れるけど、時間と技術が要る。

で、残りの全部は、この「手間」を お金 か 自由度 か 演者(実演する人) のどれかと引き換えに減らしているだけだった。だから選ぶ基準は「自分は何を差し出せるか」で決まる。

7つの手法、料金つきで (2026年7月時点)

  • Blender手付け（基準）: 0円 / 手間 最大 / 思い通り。上限は最高品質、弱点は時間と技術。
  • 文章からAI生成（HY-Motion等）: ほぼ0円 / 一瞬 / 下地向き。弱点は指と細かい制御が弱く手直しが要ること。
  • 動画から抜くAI（Move AI / DeepMotion / Plask 等）: 無料枠〜月$30・Move AIは約$0.012秒 / 生々しい。弱点は演者と動画が要ること。
  • スーツ型mocap（Rokoko / Xsens）: 機材 数十万〜 / 大量に高品質。弱点は初期投資と場所。
  • 既製ライブラリ（Mixamo 無料 / ActorCore 有料）: 0円〜 / 定番を選ぶだけ。弱点はカタログ外は作れないこと。
  • AI支援ツール（Cascadeur）: 無料〜€49月 / 綺麗に時短。弱点は買い切れないサブスク。
  • AI動画に転写（Seedance R2V 等）: 9秒で約$2.7 / 完成映像が直接出る。弱点は動きのデータが残らないこと。

どう選ぶか (逆引き)

  • 精密に作りたい → Blender手付け（Cascadeurで時短）
  • 無い動きを一瞬・無料で → 文章AIで下地→手付けで仕上げ
  • 生々しい実演 → 動画から抜くAI（無料はRokoko Vision）
  • 定番を早く → Mixamo
  • 完成したアニメ映像に → AI動画に転写（Seedance R2V）

作る側の視点: 実際に組んだパイプライン

ここからは作る人向けに、今いちばん手応えのある組み方を具体的に書く。結論は「AIに下地を出させて、仕上げの制御だけ手付けが握る」形だった。

  1. 文章からAI生成で、動きの下地を作る。ここでハマったのが、intro・キック・outroを別々に生成して繋ぐと、継ぎ目でキャラが空中で丸まって破綻したこと。連続した動き（12秒以内）は、別々に作って繋ぐより「一回で通し生成」するほうが自然だった。モデルが内部で動きを繋いでくれる。複数の候補を出して、いちばん自然なものを選ぶのがコツ。
  2. 作った動きは資産として貯めておく。同じ動きは別のキャラにも使い回せるので、プレビュー映像とデータを名前とタグで棚に並べておくと、後から「回し蹴りのモーションある?」で引き出せる。
  3. その下地を、AI動画に転写して完成映像にする。灰色のマネキンの見た目は転写せず、キャラ画像を足すと、任意のアニメキャラに同じ動きが乗る。実際にこの流れで、文章一行から作った回し蹴りを、可愛いキャラのダンスキック映像まで持っていけた。

    4. Blender手付けの「重い部分」を上流のAIが肩代わりして、細かい制御は自分が握る。全部をAIに任せると崩れるし、全部を手付けすると時間が溶ける。得意なところだけ薄く使って繋ぐのが、いちばん楽で質も出た。
